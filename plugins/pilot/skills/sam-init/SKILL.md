# SAM Project Initializer

You are initializing a new AWS SAM serverless backend project. Use `sam init` with the hello-world template as the base, then restructure and scaffold it to match our production patterns.

## Inputs

- **Project name**: `$ARGUMENTS` (required — the name for the new project, e.g. `my-api`)
- If no project name is provided, ask the user before proceeding.

## Step-by-Step Process

### 1. Run sam init

Run the following command in the **current working directory**:

```bash
sam init \
  --name $ARGUMENTS \
  --runtime nodejs20.x \
  --dependency-manager npm \
  --app-template hello-world \
  --no-tracing \
  --no-application-insights \
  --no-interactive
```

This creates a `$ARGUMENTS/` directory with SAM hello-world boilerplate.

### 2. Restructure the Lambda code

Move the generated hello-world Lambda into our standard directory layout:

```
$ARGUMENTS/
├── src/
│   └── lambda-functions/
│       └── <function-name>/    # use project name or "api" as default
│           ├── index.js
│           └── package.json
```

- Move `hello-world/app.mjs` content into `src/lambda-functions/<function-name>/index.js`, converting it to CommonJS and our handler pattern (see reference.md for the template).
- Create a minimal `package.json` with `@aws-sdk/client-dynamodb` and `@aws-sdk/lib-dynamodb` as dependencies.
- Run `npm install` inside the Lambda function directory.
- Remove the original `hello-world/` directory.

### 3. Rewrite template.yaml

Replace the generated `template.yaml` with our production pattern. See [reference.md](reference.md) for the full template. Key elements:

- **Parameters** section for environment-specific values (passed via deploy.sh)
- **Globals** with 30s timeout and CORS allow-all
- **API Gateway** resource (`AWS::Serverless::Api`) with explicit CORS config
- **DynamoDB Table** with PAY_PER_REQUEST, composite key (pk/sk), and TTL
- **Lambda Function** with:
  - `CodeUri` pointing to `src/lambda-functions/<function-name>`
  - `Handler: index.handler`
  - `Runtime: nodejs20.x`
  - Environment variables referencing DynamoDB table via `!Ref`
  - `DynamoDBCrudPolicy` for permissions
  - API events for initial routes (GET /health, GET /items, POST /items)
- **Outputs** section with API URL export

### 4. Create samconfig.toml

Create `samconfig.toml` with standard build/deploy configuration. See [reference.md](reference.md) for the template. Use the project name as the stack name.

### 5. Create deploy.sh

Create `deploy.sh` that:
- Sources `.env` for parameter overrides
- Runs `sam build && sam deploy`
- Passes parameters dynamically from environment variables
- See [reference.md](reference.md) for the template.

### 6. Create .env example

Create `.env.example` (committed) with placeholder values showing what variables are needed:

```
# Copy this to .env and fill in real values
# .env is gitignored — never commit it
FRONTEND_URL=https://your-frontend.com
```

### 7. Update .gitignore

Ensure the `.gitignore` includes:
- Standard Node/npm patterns (`node_modules/`, `.npm`, `*.tsbuildinfo`)
- SAM build artifacts (`**/.aws-sam`)
- Environment files (`.env`, `.env.test`, `.env*.local`)
- `deploy.sh`
- OS files (`.DS_Store`, `Thumbs.db`)
- Serverless directories (`.serverless/`)

Use the gitignore template from [reference.md](reference.md).

### 8. Create events/event.json

Create a sample API Gateway event for local testing.

### 9. Create CLAUDE.md

Generate a `CLAUDE.md` file that documents:
- Project overview (name, purpose placeholder for user to fill in)
- Tech stack (Node.js 20.x, AWS SAM, API Gateway, Lambda, DynamoDB)
- Architecture (API routes, DynamoDB schema)
- Project structure
- Development workflow (build, test, deploy commands)
- Code patterns (handler structure, response helpers, DynamoDB patterns)
- AI assistant guidelines

### 10. Clean up

- Remove any SAM-generated files that don't fit our structure (tests/, `__init__.py`, etc.)
- Remove the `README.md` generated by sam init (it's generic)
- Ensure `sam build` succeeds from the project root
- Ensure `sam validate --lint` passes

### 11. Final verification

Run these commands to verify the scaffold:
```bash
cd $ARGUMENTS
sam validate --lint
sam build
```

Report the final directory structure to the user and mention next steps:
1. Fill in `.env` from `.env.example`
2. Update `CLAUDE.md` with project-specific details
3. Add business logic to the Lambda handler
4. Run `sam local start-api` for local testing
5. Run `bash deploy.sh` to deploy

## Important Rules

- Always use `nodejs20.x` runtime
- Always use CommonJS (`require`), not ESM (`import`)
- Always include CORS headers in all response helpers
- Always use DynamoDB single-table design with pk/sk composite keys
- Always use `PAY_PER_REQUEST` billing for DynamoDB
- Never hardcode sensitive values — use environment variables
- Never commit `.env` or `deploy.sh`
